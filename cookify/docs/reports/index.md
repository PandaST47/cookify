# Reports — формат отчётов

Папка `docs/reports/` хранит человеко-читаемые отчёты по итерациям проекта.
Это не CHANGELOG (он генерируется из git), а документация **итерационных
изменений с точки зрения продукта**: что было сделано, зачем, какие были
trade-offs и как это выглядит в UI.

---

## 1. Когда создавать новый отчёт

- Завершён крупный этап работы (как правило, согласованный с пользователем
  батч задач).
- Есть UI-изменения, которые имеет смысл зафиксировать скриншотами.
- Решение, которое стоит объяснить будущему себе/коллеге (выбор подхода,
  отказ от альтернативы, неочевидный edge-case).

Не создаём отчёт под каждый коммит — это работа CHANGELOG.

---

## 2. Имя файла

```
docs/reports/YYYY-MM-DD-kebab-case-slug.md
```

Пример: `2026-04-25-major-update.md`.
Если за день несколько отчётов — добавляем `-1`, `-2` к slug'у.

---

## 3. Структура отчёта

Каждый отчёт обязан содержать секции **в этом порядке**:

### Заголовок (H1)
Кратко формулирует тему: «Профиль и избранное», «Рефакторинг каталога», и т. п.

### Метаданные (блок в начале)
```markdown
- **Дата:** YYYY-MM-DD
- **Автор:** Имя
- **Скоуп:** список затронутых модулей/страниц
- **Связанные коммиты:** короткие SHA
- **Ссылка на дизайн:** Figma/Storybook URL (если применимо)
```

### 1. Цель
1–3 предложения. Что должно измениться с точки зрения пользователя/продукта.
Не «я добавил компонент X», а «теперь пользователь может …».

### 2. Что сделано
Маркированный список по задачам. По каждой задаче:
- что именно изменилось,
- ключевые файлы (`src/...`) с относительным путём,
- если есть UI — ссылка на скриншот в `assets/`.

### 3. Скриншоты
Изображения положить в `docs/reports/assets/<slug>-<short>.png`.
В тексте — относительными ссылками `![alt](assets/<filename>.png)`.
Размер: viewport 1440×900, deviceScaleFactor 2 (Retina).
Капчуру делать через `docs/reports/capture.mjs` (Playwright) — скрипт
проверяется в репозиторий и переиспользуется для будущих отчётов.

### 4. Решения и trade-offs
Маркированный список «что решили / почему / от чего отказались».
Сюда попадают: выбор хранилища (localStorage vs context), отказ от
библиотеки, ручная реализация vs готовый компонент, и т. п.

### 5. Качество
Чек-лист (галочки в Markdown):
```markdown
- [x] `npm run lint` — 0 ошибок, 0 предупреждений
- [x] `npm run build` — успех, размер бандла X KB gzip
- [x] Прогон по WCAG 2.2 AA: контраст, фокус, клавиатура, screen-reader
- [x] Прогон mobile-first 360 / 768 / 1024 / 1440
- [ ] Lighthouse 90+ (если есть метрики — приложить)
```

### 6. Что осталось / Next steps
Короткий список нерешённых вопросов и следующих шагов.
Если ничего — пишем «—».

---

## 4. Скриншоты: правила

- **Источник правды** — реальная сборка, а не Figma.
- Все скриншоты делаются на одинаковой ширине (1440 px), чтобы их можно было
  сравнивать между отчётами.
- Имя файла: `<slug>-<что-показывает>.png`. Пример:
  `home-recommendations.png`, `profile-exclusions-modal.png`.
- В отчёте **обязательно** alt-текст у каждого изображения — это
  доступность и SEO одновременно.
- PNG, без сжатия с потерями. Картинки крупные, но это `docs/`, а не runtime.
